You are an expert Kong plugin developer with deep knowledge of the Kong Gateway plugin development framework (PDK), Lua programming, and session management security.

## Task
Create a production-ready custom Kong plugin named `cookie-session` that implements stateless session management entirely via encrypted, signed cookies ŌĆö with NO database or storage backend required.

## Plugin Requirements

### 1. Session Creation
- Trigger session creation when an incoming request contains the HTTP header `create_session: true`
- Generate a cryptographically random session ID (UUID v4 or equivalent, using `kong.utils` or `openssl.rand`)
- Record the current timestamp as `created_at` and `last_used_at`
- Build a session payload object:
```json
  {
    "id": "<uuid>",
    "created_at": <unix_timestamp>,
    "last_used_at": <unix_timestamp>
  }
```
- Serialize the payload to JSON, then:
  1. **Encrypt** using AES-256-GCM with a secret key from plugin config
  2. **Sign** using HMAC-SHA256 with a separate signing secret from plugin config
  3. Base64URL-encode the result
- Set the resulting value as a `Set-Cookie` response header with attributes:
  - `HttpOnly`
  - `Secure`
  - `SameSite=Strict`
  - `Path=/`
  - Configurable `Max-Age` (default: 0 for session cookie)
  - Configurable cookie name (default: `kong_session`)

### 2. Session Validation (Subsequent Requests)
When a request arrives carrying the session cookie:

a. **Tamper Detection**
   - Extract the cookie value
   - Verify the HMAC signature ŌĆö reject immediately if invalid (respond with `401 Unauthorized`)

b. **Decryption**
   - Decrypt the payload using AES-256-GCM
   - Parse the JSON ŌĆö reject with `401` if malformed

c. **Idle Timeout Check**
   - Compare `last_used_at` against the current time
   - If `(now - last_used_at) > idle_timeout_seconds`, the session is expired:
     - Respond with `401 Unauthorized`
     - Instruct the client to clear the cookie via `Set-Cookie: <name>=; Max-Age=0; ...`

d. **Rolling Session Update**
   - If validation passes, update `last_used_at` to the current timestamp
   - Re-encrypt, re-sign, and re-issue the cookie via `Set-Cookie` in the response
   - Allow the request to proceed

### 3. Failure Behaviour
- Any validation failure (bad signature, decryption error, expired session, malformed data) must:
  - Return `HTTP 401 Unauthorized` with a JSON body: `{"message": "Session invalid or expired"}`
  - Set `Set-Cookie: <name>=; Max-Age=0; Path=/; HttpOnly; Secure; SameSite=Strict` to destroy the cookie
